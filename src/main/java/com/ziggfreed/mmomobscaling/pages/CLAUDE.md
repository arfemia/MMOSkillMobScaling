# pages/ - in-game admin config UI

Router for `com.ziggfreed.mmomobscaling.pages`. The mod's ONLY interactive UI page (the HUD overlays live in `hud/`). Opened by `/mobscaling ui` (admin-gated at the command); do not add another entry point.

- **[`MobScalingAdminPage`](MobScalingAdminPage.java)** extends the engine `InteractiveCustomUIPage<EventData>`. SPEC-DRIVEN (1.0.2 rework): four `ziggfreed-common` `ui/form.SettingsForm` instances (Global, Zone HUD, Inspector HUD, and a per-world editor), each rendered from an ordered `FieldSpec` list (`buildGlobalSpecs()` / `buildZoneSpecs()` / `buildInspectorSpecs()` / `buildWorldSpecs()`) - every CONSUMED knob across all four groups is editable in-game (a knob that decodes on `WorldSettings` but deliberately applies GLOBALLY, never per-world - the per-world HUD groups' position/offsets/range/name-key-prefixes, `OpenWorld.RegionSizeChunks` - is NOT exposed on the per-world form; see the seeding bullet below). Worlds is a TWO-PANEL layout (`.ui`: `Common/UI/Custom/Pages/MmoscalingAdminPage.ui`) - a scrolling world list on the left, the add/edit editor on the right; per-world rows append the MOD-LOCAL `Pages/MmoscalingWorldRow.ui` (round-2 hardening; NOT the common `Pages/ZigListRow.ui` - see the world-row bullet below), per-knob rows append the common `Pages/ZigForm*Row.ui` templates (the NOTE template wraps multi-line, fixing the old truncated hint/note text).

## Hard rules (do NOT regress)

- **`PresetMode` is a dead knob, deliberately NOT on the UI.** Verified nothing consumes `MobScalingConfig.getPresetMode()` outside the schema (`MobScalingSettingsAsset`) and the config fold; the FieldSpec dropdown + its seed line were removed (round-2 hardening). The codec field + config fold stay (an owner may still hand-set it in `mob-scaling.json`) - do NOT re-add a UI row for it without first finding a real consumer.
- **Adding a knob is ONE spec line + ONE lang key, never a new `.ui` row.** A `FieldSpec` (id, kind, leaf path, label key, optional dropdown values) appended to the matching `buildXSpecs()` list is the whole surface change; the shared engine renders/binds/collects it. Do NOT hand-write a `.ui` row or a per-field codec key for a new knob - that regresses back to the pre-1.0.2 shape this rework replaced.
- **Global tab order is difficulty-first (round-2 hardening).** `buildGlobalSpecs()` runs `enabled` note -> Difficulty (floor/caps) -> the stat Curve (`intensity` leads it, the curve's global slope multiplier) -> rarity + distance escalation (`rarity` leads it) -> Open World last. Preserve this order when adding a new global knob; drop it into the group it semantically belongs to, not the end of the list.
- **Every leaf-bearing/toggle `FieldSpec` (all four forms) carries a `.withHint(lang-key)`.** `ui/form.FieldSpec.withHint`/`SettingsForm.applyHint` (ziggfreed-common >= the version that added them) render a `#Hint` sub-label under the row; `HEADER`/`NOTE` specs cannot carry one (`withHint` throws). A NEW knob's hint is ONE qualitative sentence (no digits restating balance defaults) at `scaling.ui.hint.<id-ish>`. A world-form field that reuses a GLOBAL label key verbatim (identical meaning, just a per-world override - most plain NUMBER/DROPDOWN leaves) REUSES that global hint key too; a TRISTATE, a pool gate, or a world-identity field (id/match/parent) has no true global equivalent (Inherit is a distinct affordance from a plain toggle) and gets its own `w_*`/`pool_*`/`world_*` key - see `buildWorldSpecs()`'s doc comment for the exact rule.
- **The per-world editor shows what a blank/Inherit field currently resolves to.** `refreshWorldHints` (called from `handleEditWorld`/`handleSaveWorld`/`handleClearWorld`) composes the spec's static hint with a computed "Inherits: {value}" line (`scaling.ui.world.inherits`) via `Message.join(staticHint, Message.raw("\n"), inheritsMsg)` for every field still blank/Inherit; an authored field shows the static hint alone. The effective value is `WorldSettingsConfig.effectiveById` (the `Parent`-merged view) falling back to the GLOBAL live `MobScalingConfig` value per leaf (`effectiveWorldDisplayValues`) - a Pool gate's global is allow-all / an empty deny list / neutral scale / zero extra slots; a per-world HUD tri-state's global is the zone/inspector enabled flag. A blank `id` (a cleared/new editor) always shows static-only - there is nothing yet to resolve.
- **The Global tab is two-panel: settings left, a live skeleton-preview right.** The right column (`#MmoscalingPreviewList`, five appended `Pages/MmoscalingPreviewRow.ui` rows) shows a PLAIN mob (no rarity/variant) run through the CURRENT Global-form difficulty stat curve at five evenly-spaced sample difficulties between the live MinCap/MaxCap; `refreshPreview` reads each field via `previewValue` (blank/invalid falls back to the live `MobScalingConfig` getter, mirroring `collectLeaves`'s NUMBER rule but degrading instead of blocking - this is read-only, never a persist) and builds a `MobScaleFold.DifficultyStatCurve` DIRECTLY (mirrors `MobScalingConfig.buildCurve`, which is package-private in `config/` and unreachable from `pages/`). Refreshed on every Global-form `field`/`press`/`saveGlobal`/`selectPreset` event (never on Zone/Inspector/World events) via a small PREVIEW-ONLY partial update - it never re-pushes the form's OWN values (that would fight the admin's typing). **Ships MULTIPLIERS-ONLY, no absolute Skeleton HP number.** An absolute HP cell (`"x2.6 (52)"`) would need the vanilla Skeleton role's base `MaxHealth` (92 in `hytale-shared-source/HytaleAssets/Server/NPC/Roles/Undead/Skeleton/Skeleton.json`), but reading a Role's declared stat requires the full `NPCPlugin`/`BuilderManager`/`BuilderSupport` build pipeline (`RoleBuilderSystem.onEntityAdd`), which needs a LIVE spawned `NPCEntity` + `Holder` to construct - there is no lighter registry/raw-JSON accessor a plugin can query standalone (confirmed against `Role`/`BuilderRole`/`BuilderSupport`/`NPCRoleAssetTypeHandler` in the shared source). Faking a holder via reflection was rejected as exactly the fragile hand-rolled hack the native-asset-first paradigm above warns against for a UI preview. Do not hardcode a base-HP constant if revisiting this - it would drift the moment the shipped role's `MaxHealth` changes.
- **World rows wrap: `Pages/MmoscalingWorldRow.ui` is a MOD-LOCAL template, not common's `Pages/ZigListRow.ui`.** Modeled on the shared row (same child ids: `#Title`/`#Sub`/`#Badge`/`#EditBtn`/`#RemoveBtn`, `$Z.@ZigRichRowBtnStyle` kept on the buttons for click sounds) but sized for this page's 300px list panel: `#Title` WRAPS (`Wrap: true`, no fixed height) so a long world id / match pattern shows two lines instead of truncating, the row itself carries the fixed `Height: 64`, and `#Badge`/`#EditBtn`/`#RemoveBtn` are narrower to leave room. `buildWorldList` (Java) is unchanged beyond the template-path constant (`ROW`).
- **`EventData` is exactly five keys: `Action`/`Tab`/`WorldId`/`Field`/`@Value`.** Every `SettingsForm` row speaks this shape (`Action="field"` + `Field`/`@Value` on a value change, `Action="press"` + `Field` on a toggle click); a hand-bound control (the preset dropdown; it is the one row NOT spec-driven, since its entries are dynamic) must emit the SAME shape, never a bespoke key. Do NOT add a per-field codec entry back onto `EventData` - that is exactly the 24-key shape this rework collapsed.
- **NEVER reopen the page.** `handleDataEvent` never calls `openCustomPage`/a page rebuild; every branch answers with a PARTIAL `sendUpdate(cmd[, events], false)` (a full `openCustomPage` resets scroll position - the bug this rework fixed). A `"field"` action is cache-only for Zone/Inspector/World (`SettingsForm.cache`, no packet at all); a GLOBAL field additionally answers with a small PREVIEW-ONLY partial update (the skeleton preview, never the form's own values - see the preview bullet above); a world-list change (save/remove) clears + re-appends + rebinds the list rows in the SAME update - the official shared-source `ChangeModelPage.buildModelList` pattern (`hytale-shared-source/HytaleServer/builtin/Model/.../ChangeModelPage.java`): `cmd.clear(sel)`, re-append rows, rebind their buttons, `sendUpdate(cmd, events, false)`.
- **Every edit persists through ONE path: `config/MobScalingOwnerWriter`** (-> the common `JsonOverrideWriter` -> the owner file `mods/MmoMobScaling/mob-scaling.json` -> `MobScalingConfig.refreshFromDisk`). The page NEVER writes the file itself and NEVER mutates `MobScalingConfig` fields directly. A `TOGGLE` spec is INSTANT-persist (never collected by `SettingsForm.collectLeaves`; the page's `handlePress` flips + saves + optionally live-applies on click, driven by a small `id -> ToggleDef` lookup table, not a chain of switch arms). Every other tab (`Global`/`Zone HUD`/`Inspector HUD`) is ONE Save button per tab: `collectLeaves(false)` (a blank NUMBER/INT is a validation ERROR on these three forms - nothing here is optional/inheritable) then `MobScalingOwnerWriter.saveLeaves(...)`. HUD + preset edits ALSO live-apply (`ZoneDifficultyHud`/`MobInspectorHud.setEnabledForAllOnline`/`refreshPositionForAllOnline`, `swapActivePreset`); after a tab save, the form RESEEDS from config + `applyValues` so fold-clamped values (e.g. `MaxCap >= MinCap`) reflect back into the UI. The SAME `MobScalingOwnerWriter` path backs the `/mobscaling` command, so the two never diverge.
- **`Enabled` needs a restart** (the zero-cost registration gate registers systems only at `setup()`): persist + flip the field but the UI shows the "takes effect on restart" note (`scaling.ui.status.saved_restart`, wired via the `enabled` toggle's `ToggleDef`). Never try to live-(un)register.
- **The per-world editor seeds from the AUTHORED body, NEVER the Parent-merged effective view** (`WorldSettingsConfig#authoredById`, NOT `foldedView()`/`effectiveById()` - those two are backed by the SAME already-Parent-merged map, so neither is safe to seed an editor from). With ~40 exposed per-world knobs, seeding the merged view and saving back would materialize every inherited leaf into the child file and silently break inheritance; `authoredById` decodes the file's own raw JSON (jar/pack/owner id-replace layering still applies, no `Parent`-chain walk), so a blank field / the Inherit tri-state round-trips as "inherit" instead. `collectLeaves(true)` (blank IS inherit, the only form where this is true) writes a NULL leaf for anything left blank/Inherit, so the saved file carries exactly the authored leaf set. The `worldId` spec's leaf path (`$Id`) is a SENTINEL popped from the collected map before every save - the filename IS the id, there is no `$Id` codec key. Save id = the World-id field, falling back to sanitized Match (both run through `WorldSettingsConfig.sanitizeFileId` - so does the self-`Parent` comparison, or a differently-spaced/cased id+Parent pair slips the check); when BOTH id and Match are blank the save is rejected (`id_or_match_required`); a blank Match with a non-blank id IS legal (a pool-only BASE file). Remove = `deleteWorldFile` (a shipped same-id file re-folds). A self-`Parent` is rejected inline; deeper cycles warn at fold.
- **Only the per-world `ZoneHud`/`InspectorHud` `Enabled` leaf is exposed - the rest of each HUD group stays GLOBAL on purpose.** `WorldSettings` decodes a full `Hud`/`InspectorHud` body (`Position`/`OffsetX`/`OffsetY`/`RangeBlocks`/the name-key prefixes) and `OpenWorld.RegionSizeChunks`, but `MobScalingConfig`/`ResolvedWorldSettings` never READ those per-world leaves - a HUD's on-screen position is a per-PLAYER-viewport concern, not a per-world one, and the region-power grid size must stay identical across worlds for region keys to compare. The world `FieldSpec` table therefore stops at `wZoneHud`/`wInspHud` (hide-only visibility); do NOT add the rest of the HUD group or `RegionSizeChunks` to `buildWorldSpecs()` - there is nothing per-world for them to control.
- **Overriding a shipped/pack world for the first time is copy-then-edit, not replace-then-edit.** `MobScalingOwnerWriter.saveWorldFile` seeds the brand-new owner file from `WorldSettingsConfig#authoredRawJsonById` (a deep clone of the shipped body's raw JSON - its `Parent`, any `$Comment`, and every leaf, exposed by the UI or not) BEFORE merging the caller's collected leaves on top; editing an EXISTING owner file skips the seed (a plain partial merge, unchanged). Without the seed step a fresh override would carry ONLY the ~40 exposed knobs, silently dropping everything else the shipped body authored (the unexposed HUD leaves above, `RegionSizeChunks`, a `$Comment`, ...) - an owner file otherwise replaces the shipped body WHOLESALE by id (layering is id-replace; inheritance is `Parent`'s job). A blanked EXPOSED field still REMOVES that leaf from the copy (a null leaf persists as a removal exactly like it would against an existing file); an unexposed leaf is never touched.
- **A `collectLeaves` validation failure names the failing field**: `scaling.ui.status.invalid_field` with a `field` param = `Message.translation(errorLabelKey)` (a NESTED client-resolved `Message`, validated against the official `PortalDeviceActivePage` precedent - `.param(key, Message)` renders correctly on `.TextSpans`, not just chat). One generic key covers NUMBER/CHANCE/INT failures (its value text explains all three forms) rather than branching to kind-specific keys.
- **Namespace every id `#Mmoscaling*`** (the client UI namespace is flat + cross-mod). CRUD-row bare ids (`#Title`/`#EditBtn`) are safe only because the indexed parent (`#MmoscalingWorldList[i]`) scopes them; the SAME rule covers the appended `ui/form` rows (`#MmoscalingGlobalForm[i] #Field`, etc.).
- **The shared settings-UI toolkit is `ziggfreed-common`** (`ui/SettingsUiUtil`, `ui/ZigRichButton`, `ui/hud/HudPosition`, `util/JsonOverrideWriter`, `Pages/ZigListRow.ui`, and as of 1.0.2 `ui/form/` - `FieldSpec`/`SettingsForm` + the five `Pages/ZigForm*Row.ui` templates) - reuse it, do not re-derive a binding/row-rendering helper here.
